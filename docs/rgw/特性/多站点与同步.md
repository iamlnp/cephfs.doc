# 1 概念  
## 1.1 顶层三层结构
### 1.1.1 Zone  
Zone = 一个 RGW 实例的 “运行环境 + 存储配置“  
包含：  
- 元数据池、数据池、索引池
- GC、日志、生命周期池
- 存储策略（placement）
- 多站点同步配置
**一个 Ceph RGW 集群至少有 1 个 zone。**

zone 最核心作用：
1. 定义 RGW 所有池的位置
2. 定义对象存在哪个 data pool
3. 定义桶索引存在哪个 index pool
4. 多站点跨区域同步的最小单位
5. 一个 zone = 一套独立的 RGW 环境

zone 信息解析：  
```bash
[root@node1 ~]# radosgw-admin zone get
{
    # 区域基本信息
    "id": "03b4c78e-c70f-4ad7-b229-608bff012f1b",     # zone 的唯一 ID（UUID）
    "name": "rgw-scale",                             # zone 名字
    "domain_root": "rgw-scale.rgw.meta:root",        # RGW 域根路径（命名空间根）
    # 系统核心池（RGW 内部用）
    "control_pool": "rgw-scale.rgw.control",         # 集群控制、同步、状态
    "gc_pool": "rgw-scale.rgw.log:gc",               # 垃圾回收（删除对象后清理）
    "lc_pool": "rgw-scale.rgw.log:lc",               # 生命周期（过期/自动删除）
    "log_pool": "rgw-scale.rgw.log",                 # 访问日志
    "intent_log_pool": "rgw-scale.rgw.log:intent",   # 多站点同步 intent log
    "usage_log_pool": "rgw-scale.rgw.log:usage",     # 用量统计日志
    "merge_pool": "rgw-scale.rgw.meta:merge",           # 元数据合并
    "tenant_tid_pool": "rgw-scale.rgw.meta:tenant.tid"  # 租户 TID
    "reshard_pool": "rgw-scale.rgw.log:reshard",     # 桶分片（reshard）任务
    "otp_pool": "rgw-scale.rgw.otp",                 # 一次性密码/双因子
    # 用户元数据池（存所有用户信息）
    "user_keys_pool": "rgw-scale.rgw.meta:users.keys",    # AK/SK
    "user_email_pool": "rgw-scale.rgw.meta:users.email",  # 用户邮箱
    "user_swift_pool": "rgw-scale.rgw.meta:users.swift",  # Swift 用户
    "user_uid_pool": "rgw-scale.rgw.meta:users.uid",      # UID → 用户信息
    "roles_pool": "rgw-scale.rgw.meta:roles",             # 角色权限
    "system_key": {
        "access_key": "",
        "secret_key": ""
    },

        {
            "key": "policy1",    # 存储策略名字（创建桶时用）
            "val": {

                    "STANDARD": {                 # 标准存储类型

                },
                "data_extra_pool": "rgw-scale.rgw.non-ec",  # EC 辅助池
                "index_type": 0,
                "cache_pool": "",
                "cloud_classes": {}
            }
        }
    ],
    "realm_id": "",
}
```

### 1.1.2 ZoneGroup  
ZoneGroup = 管理一组 Zone 的 “逻辑容器 / 命名空间 / 同步域“  
- 一个 ZoneGroup 包含 **1 个或多个 Zone**
- 同一 ZoneGroup 内的所有 Zone **共享同一个全局命名空间**（桶名、用户全局唯一）Ceph
- 同一 ZoneGroup 内的 Zone 之间 **自动双向 / 单向同步数据（多站点 / 灾备）**
- Zone 是 “数据存储单元”，真正**存储数据、定义池、处理读写**, 属于**一个 Ceph 集群**（不能跨集群）
- ZoneGroup 是 “同步与命名空间管理单元”，**不存数据**，只做**管理、同步、命名、路由**，可以**跨多个 Ceph 集群**

层级关系（从上到下）:  Realm（域） → ZoneGroup（区域组） → Zone（区域） → RGW 实例 → Pool  

ZoneGroup 信息解析：  
```bash
[root@node1 ~]# radosgw-admin zonegroup get
```

### 1.1.3 Realm  
- **最高层命名空间**
- 全局唯一：**桶名、用户在整个 Realm 内唯一**
- 管理：多个 ZoneGroup 属于同一个 Realm
- 作用：**跨地域全局统一命名**

## 1.2 同步核心角色  
1. Master ZoneGroup（主区域组）
    - 整个 Realm 里**只有 1 个**
    - 所有元数据（桶、用户、权限）**必须写主 ZoneGroup*
2. Master Zone（主区域）
    - 一个 ZoneGroup 里**只有 1 个主 Zone**
    - 元数据写入入口
    - 其他 Zone 从它同步元数据
3. Secondary Zone（从区域）
    - 只读 / 可写（取决于配置）
    - 自动从 Master Zone 同步元数据 + 对象数据
    - 可本地读、本地写（会同步回主）

## 1.3 同步两大类  
1. 元数据同步（Metadata Sync）
    - 同步：桶、用户、AK/SK、ACL、Policy、生命周期
    - 必须走 Master Zone
    - 强一致、全局唯一
2. 数据同步（Data Sync）
    - 同步：对象数据（PUT/DELETE）
    - 可**双向同步**（active-active）
    - 最终一致
## 1.4 核心同步名词  
1. Source Zone → 同步源： 数据从哪来
2. Destination Zone → 同步目标：数据同步到哪
3. Sync Policy（同步策略）
    - 同步哪些桶 / 前缀
    - 是否同步所有 Zone
    - 限流、过滤
4. Bucket Instance（桶实例）
    - 桶在**每个 Zone 里的实体**
    - 一个桶在多 Zone 存在多个 instance
    - 数据同步就是在 instance 之间复制
5. Log Sync（同步日志）
    - `log_pool` , `intent_log_pool`
    - RGW 不实时推送，而是**写日志 → 异步回放同步**
6. Tombstone（墓碑）
    - 删除对象不是真删，是写墓碑
    - 同步墓碑 = 远程删除
7. Reshard、Bucket Shard
    - 桶索引分片
    - 多站点必须保持**相同 shard 数**才能同步
