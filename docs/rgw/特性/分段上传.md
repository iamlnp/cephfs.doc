Ceph RGW 的分段上传（Multipart Upload），你可以把它理解为在大文件上传过程中，将任务拆解为多个小任务并行处理的功能。

它的核心价值在于解决了两个关键痛点：一是绕过了普通上传接口 5GB 的文件大小限制[^1]；二是将网络中断的影响从“前功尽弃”降低为“仅重传失败分片”，让大文件上传更可靠[](https://developer.aliyun.com/article/762437)。  

# 工作流程  
1. 初始化上传 (Initiation)
    - 客户端发起请求，RGW 会分配一个唯一的 Upload ID 并记录元数据（如访问控制策略），存储在 `.rgw.non-ec` 池中，作为本次上传的“凭证”[^2]。
    - **技术实现**：在代码层面，此操作由 `RGWInitMultipart::execute()` 函数处理[^3]。
2. 上传分片 (Uploading Parts)
    - 客户端将文件切分成多个分片，并**按任意顺序**并行上传。每个分片上传成功后，RGW 都会返回一个 `ETag`（分片内容的MD5校验值）作为凭证。
    - **分片处理**：如果客户端上传的分片大于 RGW 内部的默认分段大小（通常是4MB），RGW 会进一步将其切分成多个“条带”（stripes）进行存储
3. 完成上传 (Completion)
    - 上传所有分片后，客户端发送完成请求，将所有分片的 `ETag` 和分片号按顺序发给 RGW。RGW 验证无误后，会将所有分片拼接成一个完整的对象。

# 核心限制与建议  

使用时，请务必留意以下关键限制：

| 项目         | 限制                    | 说明            |
| ---------- | --------------------- | ------------- |
| **分片数量**   | **最多 10,000 个**       | 这是硬性限制        |
| **分片大小**   | 最小 **5MB**，最大 **5GB** | 最后一个分片可以小于5MB |
| **普通对象大小** | 普通上传最大 **5GB**        | 超过此大小必须使用分段上传 |

# 中断清理与垃圾回收  
果上传中断或失败，未完成的“影子数据”和临时文件需要被清理。RGW 提供了两种机制：    
- **生命周期策略**：最推荐的方式。你可以为桶配置规则，例如 `DaysAfterInitiation: 7`，让 RGW 在初始化上传后7天自动清理未完成的分片，避免手动管理的麻烦[^4]。
- **后台垃圾回收**：RGW 的后台 GC 线程会定期扫描并回收这些临时数据。如果 GC 队列发生拥塞，可能会影响系统性能，出现客户端请求阻塞或 504 错误[^5]。
